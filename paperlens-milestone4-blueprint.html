Multi-PDF Analysis Engine

This milestone transforms PaperLens from a PDF parser into a genuine study intelligence platform. A student uploads five years of question papers and gets back a ranked, deduplicated, topic-organized question bank with preparation scores — built without requiring AI for the core analysis logic. This blueprint covers every architectural decision, database schema, algorithm choice, and build order.

System Architecture

The Milestone 4 engine adds three new service layers on top of the extraction pipeline you already have. The existing system ends at the point where questions are extracted from a single PDF. Milestone 4 picks up from that point and introduces cross-paper reasoning: merging, deduplicating, ranking, and summarizing across any number of uploaded papers.

Blueprint 02 / 10

Database Design

The database schema is the single most important engineering decision in this milestone. Get this right before writing a single line of service code. The key insight is the distinction between a raw question (one occurrence in one PDF) and a question group (the canonical, deduplicated record that aggregates all occurrences). Everything — priority scores, frequency counts, year trends — is computed from the group, not from individual questions.

Architect's Decision

Do NOT store frequency counts as a column that you update incrementally. Compute them from a COUNT(*) over the question_occurrences table every time. Precomputed counts get out of sync when papers are deleted or re-uploaded. Compute on read; cache when it becomes slow (it won't for student-scale data).

Blueprint 03 / 10

Similarity Detection Strategy

This is the most technically interesting part of Milestone 4 and also the area where many beginner implementations fall apart. The core challenge: two questions about the same concept may share only 30% of the same words. A naive string-match approach will miss them entirely. But jumping straight to embeddings means introducing an external API dependency (cost, latency) for a feature that a well-tuned fuzzy matcher handles well at student-scale data volumes.

The recommended approach is a three-phase strategy where each phase is optional and builds on the last. Build Phase 1 for the MVP. Add Phase 2 once you have real test data to validate with. Reserve Phase 3 for when you have thousands of questions and Phase 2 starts slowing down.

Phase 1 — Exact Deduplication + Fuzzy Matching (Build Now)

These two steps cover 80% of the duplicate/similar-question problem for typical exam papers. They require no AI API calls, no external models, and they run fast even on modest hardware.

Step 1A: Exact Deduplication via Content Hash

Before any fuzzy work, normalize every question's text and compute a SHA-256 hash. Any two questions with the same hash are guaranteed identical. Group them instantly. Normalization steps:

  1. Convert to lowercase
  2. Strip all punctuation except parentheses around sub-question labels
  3. Collapse all whitespace (tabs, multiple spaces, newlines) to a single space
  4. Trim leading/trailing whitespace
  5. Strip common question prefixes: "Q.", "Question:", "Q1.", "(a)", etc.
Why this order?

Always run exact deduplication before fuzzy matching. This drastically reduces the number of question pairs that the slower fuzzy comparison needs to process. If you have 200 questions and 40 are exact duplicates, fuzzy matching only needs to process ~160 questions, not 200.

Step 1B: Fuzzy String Matching (RapidFuzz)

For questions that are not exact matches, use the RapidFuzz library (a fast Python implementation of the Levenshtein-based fuzzy matching algorithm). The recommended metric is token_set_ratio, not the basic ratio — it handles word order differences well, which matters for exam questions.

Fuzzy metric comparison
MetricWhat It MeasuresUse Case
ratioCharacter-level edit distance. Very sensitive to word order.Typo detection. Not good for paraphrased questions.
partial_ratioBest matching substring. Good for short strings inside long ones.Sub-question matching within parent questions.
token_sort_ratioSorts tokens alphabetically first, then compares. Handles word-order variation.Good for short questions (fewer than 10 words).
token_set_ratio ✓Splits into intersection + remainder, compares best combination. Most robust to paraphrasing.Recommended for PYQ questions. Handles "Explain X" vs "Discuss the working of X".

Recommended threshold: 82–87. Below 82, too many false positives (unrelated questions grouped together). Above 87, too many false negatives (similar questions missed). This range is a starting point — validate with your actual test data and adjust. Log all "boundary" matches (score 80–90) for manual review during testing.

Performance note: Comparing every question against every other question is O(n²). For 500 questions that's 125,000 comparisons — fast. For 5,000 questions that's 12.5 million — slow. Use RapidFuzz's process.cdist with a score cutoff and workers argument for parallel processing. This will be fine for all realistic student use cases.

Phase 2 — TF-IDF + Cosine Similarity (Add for Richer Grouping)

TF-IDF (Term Frequency–Inverse Document Frequency) converts each question into a vector based on word importance across the entire question collection. Cosine similarity then measures the angle between two vectors — independent of question length. This catches paraphrases that fuzzy string matching misses because it ignores common words ("the", "of", "what") and amplifies domain-specific terms ("deadlock", "semaphore").

Fuzzy matching vs TF-IDF cosine similarity
RapidFuzz (Phase 1)TF-IDF Cosine (Phase 2)

Recommended hybrid: run fuzzy first, flag any question pairs it didn't group but TF-IDF cosine similarity scores above 0.65 as "candidate similar — needs review". These can be auto-grouped or presented to the user as suggestions.

Phase 3 — Sentence Embeddings (Future Enhancement)

Embeddings transform a sentence into a dense vector in a high-dimensional semantic space. Two sentences with similar meaning — even with completely different words — end up close together in this space. This is the only approach that can catch truly paraphrased questions.

When Do You Actually Need a Vector Database?

You do not need a vector database for the MVP or even Phase 2. A vector database (pgvector, Chroma, Qdrant, Weaviate) is a specialized storage engine optimized for nearest-neighbor search over millions of embedding vectors. For PaperLens, the threshold for needing one is roughly:

  • Under 2,000 questions: Store embeddings as JSON/array columns in PostgreSQL. NumPy cosine similarity over the full set finishes in milliseconds.
  • 2,000–50,000 questions: Use pgvector — a PostgreSQL extension that adds a native vector column type and approximate nearest-neighbor index. No new infrastructure.
  • 50,000+ questions: Dedicated vector database (Chroma locally, or Qdrant/Weaviate hosted). This is not a student portfolio scale — don't build for it yet.
Common Beginner Mistake

Adding a vector database before you have embeddings, and adding embeddings before you have a working question bank, and building the question bank before PDF extraction is reliable. Every step in this list depends on the one before it. Build in order. Phase 1 fuzzy matching will work very well for a portfolio-scale project.

Priority Scoring Engine

The priority score is the headline feature of PaperLens — the number a student looks at to decide what to study first. It must be a composite signal, not just a raw repeat count. A question that appeared in 3 out of 5 papers, carries 10 marks, and was asked last year is far more important than one that appeared 3 times across papers from 8 years ago for 2 marks each. The score must reflect that.

The Six Factors

Priority Score Formula (0–100)
Score = (Ffreq × 0.30) + (Frecency × 0.25) + (Fmarks × 0.20) + (Fspread × 0.15) + (Fcluster × 0.07) + (Fchapter × 0.03)
F_freq
30%
Frequency: how many times the question appeared across all papers in the session. Normalized 0–100 relative to the most frequent question.
F_recency
25%
Recency: how recent was the last occurrence? Last year = 100, 2 years ago = 75, 3+ years ago = diminishing returns.
F_marks
20%
Marks weight: avg_marks / total_marks_in_paper. A 10-mark question in a 50-mark paper = 20%. Normalized 0–100.
F_spread
15%
Year spread: number of distinct years the question (or its cluster) appeared in. Wider spread = more consistently examined. Normalized 0–100.
F_cluster
7%
Cluster size: how many similar (not identical) questions are in the same similarity cluster. Bigger cluster = broader concept, more likely to reappear.
F_chapter
3%
Chapter importance: optionally boosted if the user's context marks certain chapters as high-weightage. Default = 50 (neutral) for all chapters.

Normalization Rules for Each Factor

FactorRaw ValueNormalization RuleExample
F_freq Integer count (1–N) (count / max_count_in_session) × 100 Question appeared 4 times; most frequent question appeared 5 times → F_freq = 80
F_recency Year of last occurrence max(0, 100 − (current_year − last_year) × 20) Last appeared 2024, current = 2025 → 100−20 = 80
F_marks Average marks (avg_marks / total_marks) × 100, if total_marks known; else (avg_marks / max_marks_in_session) × 100 10 marks, 70-mark paper → F_marks = 14.3
F_spread Distinct year count (distinct_years / total_years_in_session) × 100 Appeared in 3 of 5 session years → F_spread = 60
F_cluster Cluster group count min(100, (cluster_size − 1) × 25) Cluster of 4 groups → F_cluster = 75
F_chapter User-defined 0–100 or auto 50 Pass through directly User marks "Deadlocks" as high priority → 90

Priority Tier Thresholds

Score RangeTierMeaningStudent Action
85–100CriticalAppeared recently, frequently, and carries high marks. Near-certain to appear.Prepare an answer you can write in exam conditions immediately.
70–84Very HighHighly consistent pattern across multiple years.Prepare fully — strong likelihood of appearing.
50–69HighAppeared multiple times but may have gaps in recent years or lower marks.Prepare — included in core study plan.
30–49MediumAppeared once or twice, or older pattern.Review if time permits — optional depth.

Blueprint 05 / 10

Multi-PDF Processing Pipeline

Processing multiple PDFs is not simply running the single-PDF pipeline five times in a loop. The key differences are: state management across the batch, idempotency (handling re-uploads), coordination between papers before running cross-paper analysis, and user feedback during a potentially slow operation.

Session-Based Processing Model

Every multi-PDF upload is wrapped in an analysis session. The session is the unit of work. It moves through a defined state machine:

Fig 5.1 — Session state machine. Each session persists its state so the UI can poll for progress.

Step-by-Step Pipeline

01
Receive Batch Upload + Create Session
API endpoint accepts N files + optional context form data. Creates analysis_session record (status = CREATED), creates one papers record per file with status QUEUED, saves files to /uploads/session_id/.
Immediately returns session_id to frontend — no waiting for processing
02
Idempotency Check (Duplicate Paper Guard)
Before processing each paper, compute SHA-256 of file bytes. Query papers table for matching content_hash within this session. If found: skip processing, link to existing extracted questions. This prevents wasted work if a user uploads the same file twice.
Also check across sessions: if paper was already processed in a previous session, re-use extracted questions
03
Year Extraction from Filename
Parse 4-digit year from filename using regex: OS_2023.pdf → 2023, Mid_Sem_Exam_2022_OS.pdf → 2022. If no year found, set paper status to NEEDS_YEAR and ask the user to confirm/enter it before proceeding. Year is too critical for scoring to guess wrong.
If user provided year in context form, use that and skip parsing
04
Per-PDF Extraction (Existing M1–M3 Logic)
Run existing PDF extraction + OCR + question segmentation on each paper. Update papers.extraction_status as work proceeds. Store all questions in raw_questions with paper_id and year. Update session status to EXTRACTING when the first paper starts, MERGING when all finish.
MVP: run sequentially. Future: run concurrently with FastAPI BackgroundTasks or asyncio.gather
05
Normalize All Questions
Once all papers are extracted, run the normalization pipeline across all questions: lowercase → strip punctuation → collapse whitespace → strip question-number prefixes → compute SHA-256 content_hash. Store normalized text and hash back to raw_questions.
Do not normalize during extraction — always do it as a separate idempotent pass
06
Exact Deduplication → Create Question Groups
Group all raw_questions by content_hash. Each unique hash = one question_group. Create question_occurrences records linking each raw question to its group. The group's canonical_text = the longest/most complete version among its occurrences.
Update session status to ANALYZING after this step
07
Fuzzy Similarity Grouping → Create Clusters
Run RapidFuzz token_set_ratio over the set of unique group canonical_texts. Any two groups scoring above threshold (default 84) get assigned to the same similarity_cluster. One group per cluster becomes the representative (longest canonical_text). Update question_groups.cluster_id.
Store method='fuzzy' and the threshold used in similarity_clusters — important for debugging and future recalibration
08
Topic Classification
For each question group, test the canonical_text against keyword lists for each topic. If user provided chapters, build keyword lists from chapter names + common domain terms. Assign best-matching topic_id. If no match, assign a catch-all "Uncategorized" topic. Store confidence score for future manual review UI.
This is the step most improved by user context. Always run context-aware classification before generic keyword classification.
09
Priority Score Computation
For each question group: COUNT occurrences, find last_year, compute avg_marks, count distinct years, look up cluster size. Apply the 6-factor formula (see Blueprint 04). Store priority_score and priority_level enum on the group. Set score_computed_at.
Scores are always recomputed from scratch — never updated incrementally. Set a stale flag if new papers are added to the session.
10
Analytics Pre-computation + Session Completion
Run the AnalyticsService to compute all dashboard metrics (frequency distribution, chapter weightage, year trends). Cache results to analytics_cache table with computed_at timestamp. Set session status to COMPLETE. Frontend polling detects status change and loads the dashboard.
Analytics cache TTL: invalidate whenever new papers are added to the session or scores recomputed

API Endpoints for the Dashboard

Design the API to serve the dashboard efficiently. Avoid the trap of having the frontend make 8 separate API calls to load 8 panels — that causes slow load times and waterfall requests. Provide a single summary endpoint plus lazy-load endpoints for heavy/optional data. 

# Session management

GET    /api/sessions/{id}               # Status polling + summary stats

# Question bank (the main data view)
GET    /api/sessions/{id}/questions     # Paginated, with filters + sort

# Analytics (served as single precomputed payload)

Architect's Recommendation

Use the left border + badge combination as your default question card design. The left border provides instant visual scanning; the badge provides the label for accessibility, filtering, and screen readers. Add the numeric score only in the expanded detail view. This is the pattern used by Jira (issue priority), Linear (urgency), and most task management tools — it's learned behavior students will understand immediately.

Question Card Design Specification

Each question card must communicate all critical information at a glance. Proposed information hierarchy:

Collapsed Card (List View)

  • Left border: priority tier color (4px)
  • Top-right: priority badge ("Critical") + occurrence count ("×5")
  • Body: canonical question text, truncated to 2 lines
  • Footer: chapter tag + marks + question type + year range (e.g. "2020, 2022, 2024")
  • Click to expand

Expanded Card (Detail View)

  • Full canonical question text
  • Priority score breakdown (mini bar chart per factor)
  • All occurrences listed: year · marks · question number · source paper
  • Similar questions in same cluster (linked)
  • Chapter/topic with keyword-match explanation

Upload Flow UX

The upload experience must handle multi-file selection gracefully and give clear progress feedback for a potentially slow processing pipeline.

Risk Analysis

Technical Risks

RiskSeverityLikelihoodMitigation
Year not extractable from filename High Common Never guess. If regex fails, block processing and prompt the user to input the year manually. A wrong year corrupts recency scoring for the entire session.
Fuzzy threshold too low → false groupings Medium Likely during tuning Build a test suite with known-correct pairs before going to production. Log all groupings with score in range 75–90 for manual spot-check. Let users "ungroup" incorrectly clustered questions.
Priority scores feel wrong to users High Moderate Show the score breakdown in detail view (which factors contributed what). Make weights configurable via context (user can say "marks are not important for my exam"). Never present scores as authoritative.
Topic classification failures ("Uncategorized") Medium Likely without context Show "Uncategorized" prominently. Prompt user to review and manually tag if context wasn't provided. Use classification confidence score to identify low-confidence tags for user review.
Slow processing for large batches Medium Low (student scale) For MVP, synchronous processing is fine for up to ~10 PDFs. Add FastAPI BackgroundTasks for the analysis phase so the upload endpoint returns immediately. If OCR is involved, expect 15–30 seconds per page — communicate this to users upfront.
Re-upload of same paper creates duplicates Medium Common (users resubmit) Content hash check on paper upload (Step 2 in pipeline). If hash exists in this session, skip and return the existing paper record. This is a must-have guard, not optional.

Algorithm Limitations

LimitationImpactHandling Strategy
Fuzzy matching misses semantically similar questions with different vocabulary ("Explain" vs "Describe", "Compare" vs "Differentiate") Some similar questions appear as separate groups instead of clustered Acceptable at MVP. Add TF-IDF cosine similarity in Phase 2 to catch these. Until then, show "Similar questions you might also want to review" as a suggestion, not a hard grouping.
Priority scoring weights are arbitrary without real student outcome data Scores may not perfectly predict exam likelihood Present scores as "guidance" not "prediction". Add explicit disclaimer in UI: "Based on pattern analysis — not a guarantee of exam content."
Topic classification keyword matching fails for highly technical questions Questions classified as "Uncategorized" even when topic is obvious to a human Let users correct classifications. Log corrections and use them to improve keyword lists over time.
Questions with diagrams, mathematical notation, or code blocks are extracted as broken text Those questions have garbled text, grouping fails Detect suspected diagram/equation questions (short text, lots of special characters) and flag with has_special_content = true. Exclude from fuzzy matching. Show them separately in a "Review needed" section.

Scalability Concerns

For the current portfolio use case (one student, a handful of subjects, dozens of papers), scalability is not a real concern — SQLite with the right indexes will handle it without breaking a sweat. However, knowing where the walls are demonstrates senior thinking:

  • Fuzzy matching O(n²): Hits 5–10 second territory around 5,000 unique questions. Mitigation: RapidFuzz's cdist with workers=-1 (all CPU cores) pushes this to ~50,000 before becoming painful. Vector indexing resolves it beyond that.
  • Analytics queries slow down: Complex JOINs with GROUP BY across 100k+ rows get slow. Mitigation: pre-computed analytics cache (already in design). Invalidate and recompute when session changes.
  • File storage: PDFs are large. 100 users × 10 papers × 5MB average = 5GB. Local disk fine for portfolio. S3 (or Cloudflare R2 — free tier) for anything beyond that.

AI Integration Architecture Principle

Key Design Rule for AI Features

Every AI call in PaperLens must go through two guards before executing: (1) a cache check (has this exact content been processed before? Look up by content_hash), and (2) a necessity check (does rule-based logic already handle this well enough?). This design ensures your AI features enhance quality without running costs you can't control. The rule-based system is always your fallback — never build an AI feature without a non-AI fallback.
